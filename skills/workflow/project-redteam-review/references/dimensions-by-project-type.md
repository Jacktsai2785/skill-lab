# 專案型態 → 三個紅隊面向對照

Phase 0 判定型態後，挑對應的三個面向。三面向刻意設計成**互補/對立**，讓盲點不重疊。
若專案橫跨多型態（如 web app + 資料管線），挑「最能代表風險集中處」的三個，並在報告註明取捨。

| 專案型態 | 面向 ① | 面向 ② | 面向 ③ |
|---|---|---|---|
| Web app / API service | 安全與攻擊面（authn/z、注入、機密外洩） | 正確性與邊界（錯誤處理、併發、輸入驗證） | 架構與可維護性（分層、耦合、可測試性） |
| CLI / 工具程式 | 正確性與邊界（退出碼、邊角輸入、冪等） | 使用者體驗與健壯性（錯誤訊息、失敗復原） | 可維護性與精簡（重複、過度抽象） |
| 資料管線 / ETL / 爬蟲 | 資料正確性與完整性（去重、漏抓、schema 漂移） | 健壯性與可靠性（重試、部分失敗、冪等） | 效能與可擴展性（批量、I/O、瓶頸） |
| Library / SDK | API 設計與相容性（介面、破壞性變更、預設值） | 正確性與邊界（型別、錯誤語義、執行緒安全） | 文檔與可用性（範例、命名、最小驚訝原則） |
| 後端服務 / 長跑 daemon | 可靠性與容錯（健康檢查、優雅關閉、資源洩漏） | 安全與機密（連線字串、權限、稽核） | 效能與可擴展性（連線池、快取、熱點） |
| 前端 / SPA | 正確性與狀態管理（race、stale state、邊界） | 效能與載入（bundle、render、N+1 請求） | 無障礙與 UX 健壯性（錯誤態、空態、可存取性） |
| 不確定 / 混合 | 安全與機密 | 正確性與邊界條件 | 架構與可維護性 |

## 跨型態都該帶進每個面向的紅隊心態
- 不是「挑毛病」而是「想出更好的設計」——每個發現都要附**建議改法**。
- 只在 Phase 1 事實基準上提問；找不到根據的猜測不寫。
- 精簡也是一種設計品質：冗餘、重複、過度抽象都算問題。
