# 共用檢視清單（rubric）與三個審視視角

這支 skill 的三個紅隊**不分工**：三隊看同一份完整清單，各自提批判點與建議調整。
diversity 來自「**不同視角**」而非「不同範圍」——這樣三份報告才不會塌成同一份，又不會像分工那樣各自只顧一塊、漏掉跨領域問題。

涵蓋範圍永遠是**兩類都要**：① 技術健康　② 產品體驗（workflow、UI/UX、貼近使用者初衷、human-friendly）。

---

## 三個審視視角（persona，所有專案型態通用）

每個視角都要**同時涵蓋技術健康與產品體驗**，差別只在切入重心：

- **視角 A — 新手 / 終端使用者**：第一次接觸這專案。最在意 workflow 是否直覺、UI/UX、錯誤訊息與空狀態、onboarding、命名與用詞、**是否真的解決它宣稱要解決的問題（貼近初衷）**。技術上挑「會讓使用者直接踩到」的 bug 與摩擦。
- **視角 B — 重度使用者 + 維護者**：每天用、也要改它。最在意正確性、效能、可維護性、技術債、重複/過度抽象、長期操作摩擦、可觀測性。產品上挑「用久了才浮現」的體驗折損與流程繞路。
- **視角 C — 對抗者 / 守門人**：想弄壞它、或守住它。最在意安全、攻擊面、邊界與失敗模式、資料完整性、濫用情境。產品上挑「異常路徑的體驗」——出錯時使用者被丟在哪、訊息能不能行動。

三隊**獨立**作業。Phase 3 彙整時，多隊都點到只算一致性訊號；高信心仍須有直接證據、可重現行為、測試結果或 code↔文檔交叉證據。只有一隊點到則列為觀點差異，由裁決決定取捨。

## 共用檢視清單 by 專案型態

三個視角都對照同一份。依型態調整兩類的重點：

| 型態 | 技術健康重點 | 產品體驗重點 |
|---|---|---|
| Web app / API | 安全（authn/z、注入、SSRF）、正確性與併發、架構分層、可測試 | 主要 user flow 摩擦、表單/錯誤訊息、載入與回饋、貼近初衷 |
| CLI / 工具 | 退出碼、邊角輸入、冪等、可維護 | 旗標直覺性、help/錯誤訊息、預設值、最小驚訝原則 |
| 資料管線 / ETL / 爬蟲 | 資料完整性、重試/部分失敗、效能瓶頸 | 操作者體驗：可觀測性、進度/失敗回報、重跑容易度 |
| Library / SDK | 正確性、執行緒安全、API 相容性 | API 直覺性、命名、文檔與範例、錯誤訊息可行動性 |
| 後端服務 / daemon | 可靠性容錯、安全機密、效能熱點 | 運維體驗：健康檢查、log 可讀性、設定容易度 |
| 前端 / SPA | 狀態正確性、效能載入、可維護 | UI/UX、a11y、空態/錯誤態、互動回饋、貼近初衷 |
| 不確定 / 混合 | 安全、正確性、架構 | 主要使用流程、人性化程度、貼近初衷 |

## 紅隊共同心態
- 不是挑毛病，是提**更好的設計**——每點附具體建議改法。
- 以 Phase 1 **已查證事實基準**（含 Phase 0 捕捉的「專案初衷」）為共同起點；憑空想像不寫，但附有一手證據時可以提出事實基準修正。
- 「**貼近使用者初衷**」要對照 Phase 0 捕捉的專案目的 / 目標使用者 / 要解決的問題，不是審查員自己的喜好。
- 精簡也是品質：冗餘、重複、過度抽象都算問題。
- 每個發現標 `category`：`技術健康` 或 `產品體驗`，方便彙整與 GUI 分類。
